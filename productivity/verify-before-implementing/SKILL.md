# Verify-Before-Implementing

## 触发条件
用户描述目录结构、数据流程、pipeline，或需要实现涉及路径/数据组织的技能时。

## 铁律（无例外）

**Step 0 必须在 Step 1 之前完成。**

### Step 0: 验证现有结构
在写任何代码或创建任何技能之前，先检查文件系统里实际有什么：
```bash
ls /path/to/parent/           # 列出内容
find /path -type l            # 找到 symlinks
file /path/to/symlink         # 确认 symlink 目标
```

### Step 1: 与用户确认
向用户展示你发现的，并问：
> "确认一下，当前结构是 [你的观察] 对吗？我理解的是：[你的理解]。如果对我就按这个实现。"

### Step 2: 实施
仅在用户确认后。

### Step 3: 更新记忆
确认后立即写入 memory。

## 为什么这很重要
本次（2026-04-26）损失了 >1 小时，因为把 `daily/{DATE}/` 建成了**真实文件**而非**软链接**——与用户描述完全相反。用户明确说过：
- `daily/2026-04-23/new/` 应该是**软链接** → `../../../raw/REQ-xxx.json`
- `daily/2026-04-23/parsed/` 应该是**软链接** → `../../parsed/...`

但我读了 "daily" 就当真实文件处理，错误理解了结构。

## 坑：不要相信你对结构描述的解释
自然语言中的目录结构描述有歧义。实施前必须对照文件系统验证。

## 坑：跨会话 = 跨理解
之前约定的结构下次可能记错。用 `session_search` 查找之前的决策，找到后确认。
